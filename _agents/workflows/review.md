---
description: ブログ記事のレビュー・採点レポートを作成する
---

このワークフローは、ユーザーが記事のレビューを依頼した際に実行されます。（例: `/review` または `/review <記事ファイルパス>`）

対象ファイルが指定されていない場合は、ユーザーが現在開いている `src/content/blog/` 配下のファイルを対象にします。

## レビュー手順

1. 対象の記事ファイルを読み取り、全文を把握する。

2. **日本語の品質（20点）** を確認する。
   - 誤字・脱字
   - 助詞の抜け・重複
   - 文体の統一（です・ます調 or である調）
   - 1文が長すぎないか（80文字目安）

3. **技術的正確性（30点）** を確認する。
   - 記事内の重要な技術的主張について、`search_web` や `read_url_content` ツールで**公式ドキュメントや信頼できるソース**を調べて裏付けを確認する。
   - コード例が正しく動作するか検証する（`run_command` で Python 等を実行）。
   - 不正確な記述や裏付けの取れない主張があれば指摘する。

4. **構成・読みやすさ（20点）** を確認する。
   - 記事の流れが論理的か（前提→課題→解決→まとめ）
   - 見出し構成が適切か
   - コードブロック・表・図解が効果的に使われているか

5. **独自性・学習価値（20点）** を確認する。
   - 筆者自身の学びや気づきが含まれているか
   - 読者が持ち帰れるものがあるか

6. **メタデータ・フォーマット（10点）** を確認する。
   - frontmatter（title, description, pubDate, tags）が正しいか
   - `npm run format:check` を実行して Prettier 準拠か確認する。

7. 以下の形式で**採点レポート**を作成し、ユーザーに報告する。

```
## 記事レビューレポート: [記事タイトル]

| 観点                     | 配点 | 得点 | コメント |
| ------------------------ | ---- | ---- | -------- |
| 日本語の品質             | 20   | ??   | ...      |
| 技術的正確性             | 30   | ??   | ...      |
| 構成・読みやすさ         | 20   | ??   | ...      |
| 独自性・学習価値         | 20   | ??   | ...      |
| メタデータ・フォーマット | 10   | ??   | ...      |
| **合計**                 | **100** | **??** |       |

### 判定: ✅ マージ可能 / ❌ 要修正

### 指摘事項（あれば）
1. ...
2. ...

### ソース確認結果
- 「○○」の記述 → ✅ [公式ドキュメント](URL) で確認済み
- 「△△」の記述 → ⚠️ ソース未確認、要検証
```

8. **80点以上**なら「✅ マージ可能」、**80点未満**なら「❌ 要修正」と判定し、修正が必要な場合は具体的な修正案を提示する。
