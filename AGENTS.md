# AGENTS.md - Codex Review Standards

このファイルは、Codex（OpenAI）がコードレビューを自動で行う際の基準を定義します。

## プロジェクト概要

- **名前**: KaiDev Blog
- **フレームワーク**: Astro v5 + React（Islands Architecture）
- **言語**: TypeScript / CSS
- **スタイリング**: Tailwind CSS v4 + カスタムCSS
- **バックエンド**: Supabase（PostgreSQL）
- **ホスティング**: Vercel

---

## コードレビュー基準

### 1. TypeScript

- [ ] `any` 型の使用を避ける（必要な場合はコメントで理由を説明）
- [ ] 関数には適切な型アノテーションを付ける
- [ ] `interface` と `type` を適切に使い分ける
- [ ] `as` によるキャストは最小限に

### 2. React コンポーネント

- [ ] コンポーネントは小さく、単一責任に
- [ ] Props には適切な型定義を
- [ ] `useEffect` の依存配列を正しく設定
- [ ] 不要な再レンダリングを避ける（`useMemo`, `useCallback`）

### 3. Astro

- [ ] `client:*` ディレクティブは必要な場合のみ使用
- [ ] 静的生成可能なページは `prerender = true`
- [ ] Content Collections のスキーマを正しく定義

### 4. CSS / スタイリング

- [ ] CSS変数を活用してテーマ対応
- [ ] レスポンシブ対応（モバイルファースト）
- [ ] `box-sizing: border-box` の適用を確認
- [ ] 長いテキストの折り返し対応

### 5. セキュリティ

- [ ] 環境変数でシークレットを管理
- [ ] ユーザー入力のサニタイズ
- [ ] XSS対策
- [ ] CORS設定の確認

### 6. パフォーマンス

- [ ] 画像の最適化（`astro:assets`）
- [ ] 不要なJavaScriptを避ける
- [ ] コードスプリッティングの活用

## 命名規則

| 対象                       | 規則                   | 例                        |
| -------------------------- | ---------------------- | ------------------------- |
| ファイル（コンポーネント） | PascalCase             | `LikeButton.tsx`          |
| ファイル（ページ）         | kebab-case             | `blog/index.astro`        |
| 変数・関数                 | camelCase              | `handleLike`, `likeCount` |
| 定数                       | UPPER_SNAKE_CASE       | `SITE_TITLE`              |
| CSS クラス                 | kebab-case             | `.like-button`            |
| CSS 変数                   | kebab-case with prefix | `--color-accent`          |

## コミットメッセージ

```
<type>: <subject>

<body>（オプション）
```

### Type

- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント
- `style`: フォーマット（コードの動作に影響なし）
- `refactor`: リファクタリング
- `test`: テスト追加・修正
- `chore`: ビルド・ツール設定

### 例

```
feat: いいねボタンの連打防止機能を追加
fix: モバイルでの横スクロールを修正
docs: READMEを更新
```

## 記事コンテンツのレビュー基準

ブログ記事（`src/content/blog/`）の PR では、以下の観点でレビューを行い、**採点レポート**を作成する。

### レビュー観点

#### 1. 日本語の品質（20点）

- [ ] 誤字・脱字がない
- [ ] 助詞の抜け・重複がない（例：「を位置を」→「の位置を」）
- [ ] 文体が統一されている（です・ます調、あるいはである調）
- [ ] 読みやすい文の長さ（1文80文字以内を目安）

#### 2. 技術的正確性（30点）

- [ ] 記述が正確であり、誤った情報が含まれていない
- [ ] 主張に対して**信頼できるソース**（公式ドキュメント、論文、CPython実装など）が存在する
- [ ] 不確実な内容は明示的に「推測」「未検証」と記載されている
- [ ] コード例が正しく動作する

> **レビュー時のルール**: 重要な技術的主張については、レビュアーがソースを調べて裏付けを確認する。ソースが見つからない場合や誤っている場合は**指摘必須**。

#### 3. 構成・読みやすさ（20点）

- [ ] 記事の流れが論理的（前提→課題→解決→まとめ）
- [ ] 見出し構成が適切で、目次から記事全体を把握できる
- [ ] コードブロック・表・図解が効果的に使われている

#### 4. 独自性・学習価値（20点）

- [ ] 筆者自身の学びや気づきが含まれている
- [ ] 読者が具体的に何かを持ち帰れる構成になっている
- [ ] 既存記事の丸写しではなく、オリジナルの視点がある

#### 5. メタデータ・フォーマット（10点）

- [ ] frontmatter（title, description, pubDate, tags）が正しく設定されている
- [ ] Prettierフォーマットに準拠している
- [ ] 画像がある場合、altテキストが設定されている

### 採点レポート

レビュー時は以下の形式でレポートを作成する。

```
## 記事レビューレポート

| 観点 | 配点 | 得点 | コメント |
|------|------|------|----------|
| 日本語の品質 | 20 | ?? | ... |
| 技術的正確性 | 30 | ?? | ... |
| 構成・読みやすさ | 20 | ?? | ... |
| 独自性・学習価値 | 20 | ?? | ... |
| メタデータ・フォーマット | 10 | ?? | ... |
| **合計** | **100** | **??** | |

### 判定: ✅ マージ可能 / ❌ 要修正

### 指摘事項
- ...
```

> **80点以上**: ✅ マージ可能
> **80点未満**: ❌ 要修正（指摘事項を修正してから再レビュー）
