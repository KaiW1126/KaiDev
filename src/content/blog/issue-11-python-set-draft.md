## はじめに — setとは何か？

Pythonには、データを格納するための「コレクション型」がいくつか用意されています。
その中でも `set` は重複を許さない要素の集まりで、要素の順序は保証されません（未定義）。
内部的にはハッシュテーブルで管理され、存在判定が平均O(1)で行えます。

```python
my_set = {1, 2, 3, 3, 3}
print(my_set)  # {1, 2, 3} — 重複は自動で排除される
```

LeetCodeなどのアルゴリズム問題を解いていると、「この要素が含まれているかどうか」を高速に判定したい場面がたくさん出てきます。
そんな時、初学者の自分はlistを使ってO(n)の計算量で判定していましたが、listの代わりにsetを使うだけで、**計算量がO(n)からO(1)に改善される**ことが最適解として紹介されており、どういう仕組みなのか気になったので記事にします。

でも、**なぜsetはO(1)で検索できるのか？** listと何が本質的に違うのか？

この記事では、その疑問を内部構造から紐解いていきます。

---

## setとlistの違い

### 本質的な違い：「並べる」か「目印をつける」か

`list` は、要素をメモリ上に**順番に並べて格納**するデータ構造。
特定の要素を探すには、先頭から一つずつ確認していく必要があります。

`set` は、重複を許さずに存在判定が速いデータ構造。

### 操作ごとの計算量比較

| 操作                          | list | set      |
| ----------------------------- | ---- | -------- |
| 検索（`x in ...`）            | O(n) | **O(1)** |
| 追加（`append` / `add`）      | O(1) | **O(1)** |
| 削除（`remove`）              | O(n) | **O(1)** |
| インデックスアクセス（`[i]`） | O(1) | **不可** |
| 順序の保持                    | あり | なし     |
| 重複要素                      | 許可 | 不可     |

> **ポイント**: setは「検索・追加・削除」が全てO(1)で圧倒的ですが、「順序」と「インデックスアクセス」は犠牲にしています。

### その他の違い

**格納できる型の制約**

setには**ハッシュ化可能（hashable）な値**しか格納できません。setは内部でhash化して管理するため、そもそもそれに対応した値でなければ格納できない。（後述）

```python
# OK: 数値、文字列、タプルはhashable
{1, "hello", (1, 2)}

# NG: リストや辞書はhashableではないのでエラー
{[1, 2]}       # TypeError: unhashable type: 'list'
{{1: "a"}}     # TypeError: unhashable type: 'dict'
```

## なぜsetはO(1)で検索できるのか？

### そもそも内部では？

setは内部でハッシュテーブルで管理されている。

### ハッシュ関数 — 値を「住所」に変換する

ハッシュ関数とは、任意のデータを受け取って**固定長の整数値（ハッシュ値）**を返す関数です。

hashのルールとして：

- 返り値は `int`
- `a == b` なら `hash(a) == hash(b)`（等しい値は同じハッシュ値を返す）

Pythonでは `hash()` 関数で確認できます。

```python
print(hash(42))       # 42
print(hash("hello"))  # -4057395202588944075（実行環境により異なる）
print(hash((1, 2)))   # -3550055125485641917
```

このハッシュ値は、住所っぽいものを作るだけで、全部を分類できているわけではないです。（無限に近い入力から有限の整数に変換するため、被りは当然あります）
基本的にはO(1)だが、最悪O(n)になることもあります。

### 具体例：`4 in s` はどう処理される？

`s = {1, 2, 3, 4, 5}` で `4` が含まれるか判定する場合：

1. `h = hash(4)` でハッシュ値を計算する
2. そのハッシュ値からテーブル内の候補位置を見る
3. 候補位置が空なら、`4` はないと判断 → `False`
4. 候補位置に値があれば、それが `4` かどうか確かめる
5. もし衝突していて別の値が入っていたら、次の候補位置を再探索する

この再探索が繰り返されると、最悪O(n)になってしまいます。
ただし通常は衝突が少ないため、ハッシュ値を計算 → インデックスを特定 → 一致チェック、で完了します。
だから**ループ不要で一発検索 = O(1)**なのです。

### リサイズ（rehash）の仕組み

テーブルの使用率（**負荷率 = 格納数 ÷ テーブルサイズ**）が高くなると、衝突が増えて性能が落ちます。

Pythonでは負荷率が高まると、テーブルサイズを拡大して全要素を再配置（rehash）します。

リサイズ後はテーブルサイズが大きくなるため、衝突が減り、O(1)の性能が維持されます。

## LeetCodeでのsetの使い方

setの仕組みがわかったところで、実際にLeetCodeでどう使われるかを見ていきます。

### パターン1: `if x in list` → `if x in set` に置き換える

最もよくあるパターンです。listに対する `in` をsetに変えるだけで計算量が改善します。

```python
# ❌ O(n) × ループ回数 = O(n²)
nums = [1, 2, 3, 4, 5]
for x in range(10):
    if x in nums:       # listの検索 → O(n)
        print(x)

# ✅ O(1) × ループ回数 = O(n)
nums_set = set(nums)    # 一度だけO(n)でset化
for x in range(10):
    if x in nums_set:   # setの検索 → O(1)
        print(x)
```

### パターン2: 重複の検出（Contains Duplicate）

「配列に重複があるか？」という問題は、setのサイズとlistのサイズを比較するだけで解けます。

```python
def containsDuplicate(nums: list[int]) -> bool:
    return len(nums) != len(set(nums))
```

あるいは、一つずつ追加しながら重複をチェックする方法もあります。

```python
def containsDuplicate(nums: list[int]) -> bool:
    seen = set()
    for num in nums:
        if num in seen:   # O(1)で存在判定
            return True
        seen.add(num)
    return False
```

### パターン3: 2つの配列の共通要素（Intersection of Two Arrays）

set同士の積集合（`&`）を使えば、共通要素を簡単に取り出せます。

```python
def intersection(nums1: list[int], nums2: list[int]) -> list[int]:
    return list(set(nums1) & set(nums2))
```

> **共通点**: どのパターンも「ある値が存在するか？」の判定がコアにある。これがsetのO(1)検索が活きる場面です。

## まとめ — setを使いこなすために

### setが輝く場面

- **「この値、見たことあるっけ？」の判定** → `if x in seen_set`
- **重複の排除** → `list(set(data))`
- **2つのデータの共通部分・差分** → `set_a & set_b`、`set_a - set_b`

### listとの使い分け — 3つの判断基準

| 判断基準           | listを選ぶ    | setを選ぶ         |
| ------------------ | ------------- | ----------------- |
| 順序が必要か？     | ✅ 順序が必要 | 順序は不要        |
| 重複を許すか？     | ✅ 重複あり   | 重複なし          |
| 検索頻度は高いか？ | 検索は少ない  | ✅ 頻繁に検索する |
