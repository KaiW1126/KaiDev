## はじめに

システム設計を学んでいると、CDN・ロードバランサ・キャッシュ・シャーディング…といった個々の技術はそれぞれ理解できます。でも、いざ「全体としてどう繋がっているのか」を聞かれると、頭の中の地図がぼやける――自分がまさにそうでした。

この記事では、**部品を1つずつ説明するのではなく、1本のリクエストを主人公にして、それがどのサービスをどの順番で通っていくのか**を追いかけます。

```
ブラウザでURLを入力
  ↓
画面に結果が表示される
```

この「↓」の中で何が起きているのか。それを、データベースのディスクに落ちる瞬間まで分解していきます。

---

## 全体像：リクエストが辿る道のり

まず全体像です。

```
[クライアント / ブラウザ]
  │ ① URL入力
  ▼
[DNS]                      ドメイン名 → IPアドレス
  │ ② IP解決
  ▼
[CDN / エッジ]              静的コンテンツはここで返る
  │ ③ キャッシュミスなら奥へ
  ▼
[ロードバランサ (LB)]       どのサーバーに振るか決める
  │ ④ 振り分け
  ▼
[API Gateway]              認証・レート制限・ルーティング
  │ ⑤ 通過したものだけ
  ▼
[アプリケーションサーバー]    ビジネスロジック
  │ ⑥ データが必要
  ▼
[キャッシュ (Redis等)]       あればここで返る
  │ ⑦ キャッシュミスなら
  ▼
[データベース]              クエリ実行
  │ ⑧ メモリになければ
  ▼
[ディスク / ストレージ]       最終的にデータが眠る場所
```

ポイントは、**各層が「自分で返せるなら返す、無理なら奥へ渡す」というバケツリレーになっている**ことです。奥に行くほど高コスト・低速になるので、できるだけ手前で返すのが全体の設計思想になります。

以降、この①〜⑧を順番に歩いていきます。

---

## ① クライアント / ブラウザ

すべての出発点。ユーザーが `https://example.com/feed` のようなURLを入力します。

ブラウザがまず行うのは、**「このドメインのサーバーは、世界のどこにいるのか（＝IPアドレス）」を調べる**ことです。`example.com` という名前のままでは通信できないので、住所（IP）に変換する必要があります。

この変換を担うのが次の DNS です。

> 補足：ブラウザ自身もキャッシュを持っています。一度見たページのHTMLや画像、解決済みのIPアドレスはローカルに保存され、次回はネットワークに出る前に再利用されます。「一番手前のキャッシュ」はブラウザの中にある、と覚えておくと全体像が掴みやすくなります。

---

## ② DNS — 名前を住所に変える

**DNS（Domain Name System）** は、`example.com` のようなドメイン名を `93.184.216.34`（あくまで一例です）のようなIPアドレスに変換する、いわばインターネットの電話帳です。

```
ブラウザ「example.com の住所を教えて」
  ↓
DNS「93.184.216.34 です」
  ↓
ブラウザ「ありがとう、そこに繋ぎます」
```

### 解決の流れ（ざっくり）

```
ブラウザキャッシュ
  ↓ なければ
OSキャッシュ
  ↓ なければ
リゾルバ（ISPなど）
  ↓ なければ
ルートサーバー → TLDサーバー → 権威サーバー
```

ここでも「手前にキャッシュがあれば、奥まで聞きに行かない」という同じパターンが現れます。ブラウザやOSだけでなく、**リゾルバ自身もキャッシュを持っている**ため、多くの問い合わせはルートサーバーまで辿らず途中で解決します。DNSの結果には **TTL（生存期間）** が設定されていて、その間はキャッシュが再利用されます。

### 設計上のうれしさ

DNSは単なる名前解決だけでなく、**最初の振り分け装置**としても使えます。

```
- 地理的に近いサーバーのIPを返す（GeoDNS）
- 複数IPを順番に返して負荷分散（DNSラウンドロビン）
- 障害中のサーバーをIP一覧から外す（フェイルオーバー）
```

つまり「どのデータセンターに繋ぐか」という最初の交通整理が、ここで始まっています。AWSでこの役割を担うのが **Route 53** で、GeoDNSやヘルスチェックによるフェイルオーバーもここで設定します。

---

## ③ CDN / エッジサーバー — 手前で返せるものは返す

IPが分かったら、ブラウザはそのアドレスに接続します。多くの構成では、ここで最初に受け止めるのが **CDN（Content Delivery Network）** です。

CDNは世界中に分散した **エッジサーバー** 群で、画像・CSS・JavaScript・動画といった静的コンテンツを、ユーザーの地理的に近い場所から配信します。

```
ユーザー（東京）
  ↓
東京のエッジサーバー  ← キャッシュがあればここで完結！
  ↓ なければ
オリジンサーバー（遠くのデータセンター）
```

### ここで返せるもの / 返せないもの

| 返せる（静的・誰でも同じ） | 奥へ渡す（動的・人ごとに違う） |
| -------------------------- | ------------------------------ |
| 画像・動画・フォント       | ログイン後のマイページ         |
| CSS・JavaScript            | 在庫数・リアルタイム価格       |
| PDF・アイコン              | 個人情報を含むレスポンス       |

静的コンテンツが**エッジで返れば、ここから先（LB・アプリ・DB）には一切負荷がかかりません**。これがCDNの最大の価値です。逆に、人ごとに中身が変わる動的なデータはキャッシュできないので、リクエストは奥へと進みます。AWSでは **CloudFront** がこのCDNにあたり、オリジンには **S3**（静的ファイル置き場）やロードバランサを指定します。Railsアプリでも、`public/` 以下のアセットやActive Storageの画像をS3+CloudFront経由で配信するのが定番の構成です。

---

## ④ ロードバランサ — どのサーバーに渡すか

CDNで返せない動的なリクエストは、いよいよオリジン（自分たちのシステム）に届きます。最初に出迎えるのが **ロードバランサ（LB）** です。

大規模サービスではアプリケーションサーバーが1台ではなく何十台もあります。LBは、来たリクエストを**どのサーバーに振り分けるか**を決める交通整理役です。

```
                  ┌→ App Server 1
ロードバランサ ───┼→ App Server 2
                  └→ App Server 3
```

### 振り分けのルール（代表例）

```
ラウンドロビン   : 順番に均等に振る
最小コネクション : 今いちばん空いているサーバーへ
IPハッシュ       : 同じユーザーは同じサーバーへ
```

### LBがもたらすもの

```
- 負荷分散：1台に集中させない
- 可用性：死んだサーバーを自動で振り分け対象から外す
- 水平スケール：サーバーを足すだけで処理能力が上がる
```

LBがいるおかげで、アプリサーバーは「いつでも増やせる・減らせる・落ちてもいい」存在になります。これを成立させるため、**セッションなどの状態はサーバー内に持たず、外（Redisなど）に出す**のが定石です（ステートレス化）。AWSでは **ELB（Elastic Load Balancing）**、特にHTTP/HTTPS向けの **ALB（Application Load Balancer）** がこの役割を担います。Railsの場合、ALBの後ろに複数のアプリサーバー（PumaやUnicorn）を並べ、`config/environments/production.rb` でセッションストアをcookieやRedisにして状態を外出しします。

---

## ⑤ API Gateway — 門番

リクエストはアプリサーバーに渡る前に、**API Gateway** という門を通ることが多いです。LBが「どのサーバーへ」を決めるのに対し、API Gatewayは「そもそも通してよいリクエストか」を判断します。

```
リクエスト
  ↓
[API Gateway]
  ├─ 認証・認可        （あなたは誰？通っていい？）
  ├─ レート制限        （短時間に叩きすぎていない？）
  ├─ ルーティング      （どのマイクロサービスへ？）
  ├─ リクエスト変換    （形式を整える）
  └─ ログ・監視        （記録を取る）
  ↓ すべてクリアしたものだけ
アプリケーションサーバー
```

### なぜ門番を1か所に集めるのか

認証やレート制限を各サービスに個別実装すると、重複だらけになり、抜け漏れも生まれます。**横断的な関心事（認証・制限・ログ）を入り口に集約する**ことで、奥のサービスは本来のビジネスロジックに集中できます。

ここで前回までに学んだ **レート制限** が効いてきます。`429 Too Many Requests` を返してリクエストを弾くのも、多くはこの層の仕事です。

```
ユーザーID / IP / APIキー でカウント
  ↓
上限超過なら HTTP 429 を返す（アプリには届かせない）
```

門番を通過できなかったリクエストは、ここで引き返します。奥のサーバーやDBは守られたまま、です。AWSでは **API Gateway** がこの役割を専門に担い、認証（Cognitoとの連携）・スロットリング・ルーティングをマネージドで提供します。一方Railsでは、Gateway層をフルに立てずに、認証は `before_action` でのトークン検証、レート制限は `rack-attack` といったRackミドルウェアで、アプリの入り口に同等の機能を持たせることもよくあります。

なお、この層の位置はLBの前後どちらもあり得ます。マネージドなAWS API GatewayはLBの前段に置かれることが多く、Kongなどの自前Gatewayはアプリの手前（記事の図の位置）に置かれることが多い、という違いがあります。

## ⑥ アプリケーションサーバー — ロジックの中心

門を通り抜けたリクエストが、ようやく **アプリケーションサーバー** に到達します。ここがビジネスロジックの本体です。

```
リクエストを解釈する
  ↓
必要なデータを集める（キャッシュ・DB・他サービス）
  ↓
ロジックを適用する（計算・整形・検証）
  ↓
レスポンスを組み立てて返す
```

ここで重要なのは、アプリサーバーは**自分でデータを持っていない**ということです。ユーザー情報も投稿も在庫数も、すべてキャッシュやDBに問い合わせて取ってきます。

そして問い合わせるとき、いきなりDBには行きません。まずキャッシュに聞きます。Railsでいえばこの層がまさに **Railsアプリ本体**（コントローラ・モデル・ビュー）で、Pumaなどのアプリケーションサーバーが、リクエストごとにRailsアプリ（コントローラ等）を実行してレスポンスを返しています。AWS上ではこれを **EC2**、コンテナなら **ECS / EKS / Fargate**、関数単位なら **Lambda** といった選択肢の上で動かします。

## ⑦ キャッシュ — DBの手前の関所

**キャッシュ（Redis / Memcached など）** は、よくアクセスされるデータをメモリ上に置いておく高速なストアです。アプリサーバーとDBの間に立ちます。

```
アプリ「ユーザー123の情報ちょうだい」
  ↓
キャッシュ
  ├─ ある（キャッシュヒット） → 即返す！DBに行かない
  └─ ない（キャッシュミス）   → DBに取りに行く → 次回のためにキャッシュに保存
```

DBへのアクセスはメモリ参照に比べて桁違いに遅いので、**読み取りの大半をキャッシュで吸収できれば、DBの負荷は劇的に下がります**。

### キャッシュの代表的な使い方

```
- 読み取りが多く、変更が少ないデータ（ユーザープロフィール等）
- 計算コストの高い集計結果
- セッション情報（ステートレス化のために外出しした状態）
```

ただしキャッシュには**鮮度の問題**があります。元データが変わってもキャッシュが古いままだと、ユーザーに古い情報を返してしまう。そのため **TTL（有効期限）** を設定したり、更新時にキャッシュを消す（無効化）といった工夫が必要になります。AWSではマネージドな **ElastiCache（Redis / Memcached）** がこの層にあたります。Railsからは `Rails.cache`（`config.cache_store = :redis_cache_store`）や、ビューの一部をまるごとキャッシュするフラグメントキャッシュを通して、このストアを利用します。

ここまでで「手前で返す」関所をいくつも越えてきました。それでもデータが見つからない――そのときだけ、リクエストは最後の砦に向かいます。

## ⑧ データベース と ディスク — 最終到達点

キャッシュミスしたリクエストは、ついに **データベース** に到達します。ここが、データが永続的に保存されている場所です。

### DBの中でも「メモリ → ディスク」の二段構え

実はDBの内部でも、これまでと同じパターンが繰り返されています。

```
クエリが来る
  ↓
バッファプール（DBのメモリ上のキャッシュ）
  ├─ ある → メモリから返す（速い）
  └─ ない → ディスクから読む（遅い）→ メモリに載せる
  ↓
ディスク / ストレージ  ← データが最終的に眠る物理的な場所
```

つまり、リクエストの旅の本当の終着点は **ディスク（SSD/HDD やクラウドストレージ）** です。ここまで来て初めて、物理的な記憶媒体からデータが読み出されます。

### DBが詰まりやすいポイント

DBは多くのシステムで最終的なボトルネックになります。

```
- N+1クエリ（ループの中で何度もクエリ）
- インデックス不足（全件スキャン）
- 重いJOIN・不要なSELECT
- 書き込みの競合・ロック
```

### DBをスケールさせる手段

読み書きの負荷が増えてきたら、DBを分散させます。

```
リードレプリカ : 読み取り用のコピーを増やして分散
シャーディング : データを複数DBに分割（user_id等で分ける）
パーティション : 1つのテーブルを内部的に分割
```

これは「⑥アプリの水平スケール」と同じ発想を、DB層に適用したものです。AWSではリレーショナルDBを **RDS**（PostgreSQL / MySQL等）や **Aurora** で運用し、リードレプリカもマネージドで増やせます。最終的にデータが眠るディスクは、RDSなら **EBS**（DBインスタンスにぶら下がるブロックストレージ）、Auroraなら独自の分散ストレージ（複数AZにまたがる専用設計）であり、画像などのオブジェクトは **S3** に保存します。Railsからは、これらDBの違いを意識せずActive Record（`User.where(...)` のようなクエリ）越しに扱えるのがポイントです。

## 全体を貫く3つの原則

ここまで①〜⑧を歩いてきました。バラバラに見えたコンポーネントですが、実は**同じ原則が層を変えて何度も現れている**ことに気づきます。

### 原則1：手前で返せるものは手前で返す

ブラウザキャッシュ → CDN → アプリキャッシュ → DBバッファプール。どの層も「自分で返せるなら奥に行かせない」。奥に行くほど遅く・高コストなので、いかに手前で食い止めるかが性能の鍵です。

```
ブラウザ < CDN < キャッシュ < DBメモリ < ディスク
（左ほど速く・安い、右ほど遅く・高コスト）
```

### 原則2：状態を外に出してスケールさせる

LBで増やせるアプリサーバーも、DNSで振り分けるデータセンターも、「個々が状態を持たない」から自由に増減できます。状態はキャッシュやDBという**専用の場所**に集約する。

### 原則3：横断的な関心事は入り口に集める

認証・レート制限・ログといった「全リクエスト共通の処理」は、API Gatewayという門に集約する。奥のサービスは本業に専念できます。

## まとめ

1本のリクエストの旅を振り返ります。

```
クライアント   : URL入力、最初のキャッシュ
DNS            : 名前 → IP、最初の振り分け
CDN            : 静的コンテンツを手前で返す
ロードバランサ : どのサーバーへ振るか
API Gateway    : 認証・レート制限の門番
アプリサーバー : ビジネスロジック（状態は持たない）
キャッシュ     : DBの手前で読み取りを吸収
DB / ディスク  : データが最終的に眠る場所
```

- システムは、各層が **「返せるなら返す、無理なら奥へ渡す」バケツリレー**で構成されている
- 奥に行くほど遅く・高コストなので、**いかに手前で返すか**が全体の性能を決める
- 個々の技術（CDN・LB・キャッシュ…）は、この **流れの中のどの位置にいるか** で役割が決まる

個別の技術を「点」で覚えるのではなく、**リクエストの流れという「線」の上に並べて理解する**。そうすると、「なぜこの技術がここにあるのか」が腑に落ちて、設計の全体像が一枚の地図として見えてきます。
